# File Sync System

LBM keeps its task data in two places that work together:

- **`localStorage`** â€” where the live app reads and writes tasks in the browser
- **`data/tasks.json`** â€” a JSON file checked into git, used as a baseline and for AI-assisted task creation

The sync script (`scripts/sync-tasks.js`) bridges these two: it merges `tasks.json` into `data/project-data.js`, which is the seed data the app loads on first run or after a reset.

---

## Two Ways to Add Tasks

### 1. In the browser UI (you)

Click the **+** button (or press **N**), fill in the form, and save. The task is written to `localStorage` immediately. Nothing else needs to happen â€” the app is live.

This is the standard path for day-to-day task management.

### 2. Via `tasks.json` (Claude Code)

Claude Code edits `data/tasks.json` directly, then runs the sync script to push the new tasks into `data/project-data.js`. You reload the browser tab and the tasks appear.

This path is used when:
- Claude is creating tasks on your behalf during a session
- You want AI-created tasks to be version-controlled in git from the start
- You're seeding a fresh copy of LBM with a predefined task list

---

## How Claude Adds a Task (Step by Step)

1. **Read `data/tasks.json`** â€” find the highest existing ID and increment it
2. **Append the new task object** to the `tasks` array in `data/tasks.json`
3. **Run the sync script:**
   ```
   node scripts/sync-tasks.js
   ```
4. **Tell you to reload the browser tab** â€” the task will appear in the app

The sync script validates the JSON before touching `project-data.js`, so a malformed edit will print an error and leave everything unchanged.

---

## How the Two Sources Coexist

| Source | Role | Wins whenâ€¦ |
|---|---|---|
| `localStorage` | Live task state | Always â€” the app reads from here at runtime |
| `data/tasks.json` | Seed + AI baseline | On first load, or after "Reset to Seed" |
| `data/project-data.js` | Compiled seed | Generated by the sync script from `tasks.json` |

**The merge is additive.** The sync script never deletes tasks. If a task exists in `project-data.js` but not in `tasks.json`, it is preserved. If a task exists in both, `tasks.json` wins (it overwrites by ID).

**`localStorage` is never touched by the sync script.** It only updates `project-data.js`. Your live edits in the browser are always safe.

> **When does `project-data.js` affect the live app?**  
> Only when you do a **Reset to Seed** (via â“˜ â†’ Reset), or open LBM in a browser that has no `localStorage` data for this copy yet.

## When to Use Each Method

| Situation | Method |
|---|---|
| Adding a task during normal work | Browser UI |
| Claude creates tasks during a dev session | tasks.json + sync |
| Seeding a fresh LBM copy with predefined tasks | tasks.json + sync |
| Bulk-editing or reorganising the backlog | tasks.json + sync |
| Quick one-off tasks with no git record needed | Browser UI |

## `tasks.json` Field Reference

These are the fields the sync script and the app both understand. All fields are optional except `id` and `title`.

| Field | Type | Notes |
|---|---|---|
| `id` | string | Unique identifier. Format: `LBM-001`, `MCC-001`, etc. Must be unique across all tasks. |
| `title` | string | The task name. Required. |
| `notes` | string | Plain-text summary (shown in the task list). |
| `body` | string | Rich HTML content (shown in the detail panel editor). Optional. |
| `lane` | string | Kanban lane. One of: `newly-added-or-updated`, `backlog`, `processing`, `on-hold`, `in-progress`, `completed`, `archived`. Defaults to `newly-added-or-updated`. |
| `urgency` | number | 1â€“5 scale. 5 = drop everything. |
| `value` | number | Business value estimate. Higher = more impactful. |
| `priority` | string | `P0`â€“`P3`. Informational label. |
| `area` | string | Which part of the project this belongs to (e.g. `product`, `docs`, `ui-ux`, `platform`, `release`). |
| `source` | string | Where this task came from: `user-requested`, `recommended`. |
| `recommendedBy` | string | Name of the person or model that suggested this task. |
| `references` | array | File paths or doc links relevant to this task. |
| `status` | string | Legacy field (`todo`, `done`). Prefer `lane` for new tasks. |
